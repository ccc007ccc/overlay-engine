## 0. 当前出发点（必须先读）

- 仓库已在 `v0.7-final` tag（位于旧 main）
- v0.7 的 in-process 实现完整可用（Phase 1-5 全 SHIPPED）：
  - Phase 1：vector 基本图元（line / rect / rounded / ellipse / clip / transform）
  - Phase 2：bitmap 资源系统 + 画布 resize（WIC 解码 / create_texture / update_texture / draw_bitmap，含 RGBA8 swizzle、ABA slot table 1024 容量）
  - Phase 3：本地视频（Media Foundation IMFSourceReader + NV12→BGRA）
  - Phase 4：屏幕捕获（WGC）
  - Phase 5：path + 渐变
- **已视觉验证**：phase 1+2 在 Game Bar widget 实测，画面正确
- 现状架构是 **in-process DLL**：`renderer.dll` 被 host 进程加载，host 自己创建 swap chain、自己挂 DComp。这条路走通了，但**第三方程序无法调用渲染**，且**多程序共画一个画布做不到**——这就是 v1.0 要解决的问题。

---

## 1. v1.0 要解决的问题（不可议）

把 renderer 从 in-process DLL 改成**独立进程**（Core），让任意第三方程序都能通过 IPC
画东西到任意 monitor 上，monitor 也通过 IPC 拿画面挂屏。

```
                +-------------------+
                |       Core        |   ← 独立 OS 进程，独占 D2D / D3D / DComp
                |  渲染核心 server  |
                +---+-----------+---+
                    |           |
        Producer ---+           +--- Consumer
        (业务程序)               (Monitor)
        创建画布、推命令          被动接受被 attach 的画布、挂屏显示
```

---

## 2. 已敲定的架构决策（**不可议**）

### 2.1 底层渲染保留 D2D
- 不换 Vulkan / Skia / wgpu。
- **理由**：
  1. **DComp 只接受 IDXGISwapChain**，Vulkan 的 VkSwapchainKHR 不能挂 DComp。Vulkan/Skia 接 DComp 都要走"渲到 D3D 共享纹理→包装成 DXGI swap chain→挂 DComp"路径，这部分代码本身就是 Windows 独占的，跨平台收益打折。
  2. D2D 自带 DirectWrite（文字排版）+ WIC（图像解码）+ path geometry + stroke style + brush，自己用 Vulkan 重写是 3-6 个月级别的工程量。
  3. v0.7 已经验证过 D2D 选型 OK，重构时整套 D2D 命令实现可以**整体搬到 Core 进程**复用。
- **代价**：v1.0 = Windows 独占。可接受，因为 widget = Game Bar = Windows 独占的 host，v1.0 范围内没有非 Windows 的 monitor 计划。
- 跨平台是 v2.0+ 才考虑的事，到时候考虑换 Skia（不要 Vulkan，自己造 2D 引擎是坑）。

### 2.2 Core 独立进程
- Core 是一个 OS 进程，独占 ID3D11Device + ID2D1DeviceContext + DComp resources。
- 所有 producer / consumer 通过 IPC 跟它通信。
- Core 进程崩溃 → 客户端不能跟着崩；客户端断线 → Core 不能内存泄漏。

### 2.3 Producer-Owned Canvas（**用户拍板的核心模型**）
- **Core 不预创建任何画布**，启动后空载等待客户端连接。
- **画布由 producer 创建**：尺寸、格式、名字、显示在哪些 monitor 上，全部由 producer 决定。
- **Core 是机制不是策略**：只负责路由 producer 命令到画布、把画布的 surface 广播给 producer 指定的 consumer。Core 不参与"哪个画布该显示"的决策。
- **画布所有权 = 创建者**：producer 退出（IPC 断开）→ Core GC 销毁画布 + 通知所有 consumer detach。
- **Consumer 完全被动**：自己不能创建画布、不能查询有哪些画布存在、不能主动 attach。Consumer 只能注册自己 → 等 Core 推 "你被某个 producer attach 了，这是 surface handle" 事件 → 把 surface 挂到自己的可视区域。
- **多 monitor 显示同一画布**：producer 调 `attach_consumer(canvas_id, consumer_id)` 多次即可，Core 把同一个 surface handle 复制给多个 consumer。
- **多 producer 共写一个画布**：默认不允许。owner 可显式 `grant_write(canvas_id, peer_pid)` 授权别的 producer 写。

### 2.4 widget 路线必须能接入（**红线**）
- Game Bar widget = UWP MSIX 进程，渲染必须挂到 DComp visual tree，不能用普通 HWND。
- v1.0 必须支持 widget consumer：把 Core 的 swap chain 通过 DComp 跨进程 surface handle 给 widget 进程，widget 用 `compositor.CreateSurfaceFromHandle(handle)` 挂自己的 visual tree。
- 这条路如果走不通，v1.0 整个方向作废。**所以 v1.0 spec 起草第一件事就是验证 DComp 跨进程 surface 在 UWP 进程里能不能 work**——这是技术 spike，必须先做。

## 3. 三方接口规范（草案，spec 起草时细化）

### 3.1 通用
- Core 监听位置：命名管道 `\\.\pipe\overlay-core` + 环境变量 `OVERLAY_CORE_PIPE` 覆盖
- 协议头：每条 IPC 消息带 magic + version + opcode + payload_len
- capability negotiation：客户端连接时声明自己是 producer / consumer，Core 返回支持的 opcode 列表
- 心跳：客户端 5s 一个 ping，Core 10s 没收到就认为断线

### 3.2 Producer 端 API

```
# 画布生命周期
create_canvas(width, height, format, name?) -> canvas_id
destroy_canvas(canvas_id)
resize_canvas(canvas_id, width, height)

# 命令提交（**必须批量**，不能单条命令一次 IPC）
submit_frame(canvas_id, command_buffer_offset, command_buffer_len, frame_id) -> ok
request_frame_sync(canvas_id, frame_id)   # 等渲染完成

# 资源
create_bitmap(canvas_id, w, h, format) -> bitmap_id
update_bitmap(bitmap_id, shared_mem_offset, len, stride, format)
load_bitmap_from_memory(canvas_id, bytes_shared_mem, len) -> bitmap_id
destroy_bitmap(bitmap_id)

# Consumer 路由
attach_consumer(canvas_id, consumer_id)         # 把画布显示到某个 monitor
detach_consumer(canvas_id, consumer_id)
list_consumers() -> [consumer_id, metadata...]   # 查可用 monitor 列表

# 共享写权限（默认独占）
grant_write(canvas_id, peer_producer_id)
revoke_write(canvas_id, peer_producer_id)
```

### 3.3 Consumer (Monitor) 端 API

```
# 注册
register_monitor(metadata: {name, kind: "widget"|"desktop-window"|..., max_w, max_h}) -> consumer_id
unregister_monitor(consumer_id)

# 被动事件流（Core push）
on_canvas_attached(canvas_id, surface_handle, w, h, format)    # 被 producer 挂上来
on_canvas_detached(canvas_id)                                   # producer detach 或退出
on_canvas_resized(canvas_id, new_w, new_h)
on_canvas_frame(canvas_id, frame_id)                            # 一帧渲染完成（用于 perf 跟踪）

# Consumer 反馈
report_viewport(canvas_id, vp_x, vp_y, vp_w, vp_h)              # 我实际显示画布的哪块矩形
report_actual_size(canvas_id, phys_w, phys_h)                    # 我的物理像素大小
```

### 3.4 画布生命周期完整图

```
producer A 启动 → connect Core → register_producer
                ↓
producer A: create_canvas(1920, 1080, BGRA8) → canvas_id=42
                ↓
producer A: list_consumers() → [{widget, ...}, {desktop-window, ...}]
                ↓
producer A: attach_consumer(42, widget_id)
                ↓
Core push to widget: on_canvas_attached(42, surface_handle, 1920, 1080)
widget: compositor.CreateSurfaceFromHandle(surface_handle) → 挂 visual tree
                ↓
producer A: 每帧 → 把命令写 shared mem buffer → submit_frame(42, offset, len, frame_id)
Core: 解码命令 → D2D 渲染到 canvas 42 的 swap chain → DComp 自动 push 到 widget 屏幕
                ↓
producer A 退出（IPC 断开）
Core: GC canvas 42 → push on_canvas_detached(42) 给 widget
widget: 卸下 visual
```

## 4. 关键技术点（spec 起草时必须搞明白）

### 4.1 DComp 跨进程 surface 共享（**v1.0 第一个技术 spike**）

Windows 原生支持。流程：
- Core 端：每个 canvas 创建 IDXGISwapChain（独立 D3D11 device 渲染）
- Core 端：用 `DCompositionCreateSurfaceHandle()` 拿一个 NT handle
- Core 端：通过 `DuplicateHandle()` 跨进程复制句柄给 monitor 进程
- Monitor 端（widget）：调 `Windows.UI.Composition.Compositor.CreateSurfaceFromHandle(handle)` 拿到 ICompositionSurface
- Monitor 端：CreateSurfaceBrush + CompositionSpriteVisual + ElementCompositionPreview.SetElementChildVisual 挂上去（跟 v0.7 widget 现有逻辑同）

**待验证**：这条路在 UWP MSIX 沙盒进程里能不能 work（capability、句柄继承、UWP container 限制）。**v1.0 第一周必须做这个 spike**，走不通整个方向作废。

参考：Win32 DCompositionCreateSurfaceHandle、Windows.UI.Composition.Interop.ICompositorInterop（v0.7 widget 端 CompositionPump.cs 已经在用 ICompositorInterop QI compositor）。

### 4.2 命令流批量序列化（**必须，不能单条 IPC**）

- 每帧 100 条 draw 命令，60fps = 6000 命令/秒。每条一次 IPC = 性能崩。
- Producer 客户端 SDK 维护一个 ringbuffer（shared memory file mapping）
- 每条 cmd_* 调用 = SDK 把 opcode + 参数序列化追加到 ringbuffer，**不触发 IPC**
- Producer 调 `commit_frame()` = 一次 IPC 通知 Core "buffer offset X..Y 是这一帧"
- Core 解码 ringbuffer，按 opcode dispatch 到 D2D 调用
- 命令编码：固定 16-bit opcode + 可变参数（per-opcode 定长）
- 跟 v0.7 的 cmd_clear / cmd_fill_rect / ... 函数列表一一对应转 opcode

### 4.3 Bitmap 资源跨进程

- PNG/JPG bytes / RGBA pixel data 走 shared memory（不要走 named pipe，几 MB 数据 IPC 太慢）
- bitmap handle = (canvas_id, slot_id, gen)，跟 v0.7 的 BitmapHandle 设计延续
- shared memory 段命名：`overlay-core-bitmaps-<pid>`，按需扩容
- update_bitmap 时 producer 写 shared mem，Core 收到 IPC 通知后从 shared mem 读 → D2D bitmap

### 4.4 错误处理 / 崩溃恢复

- Core 崩溃：客户端 IPC 收 EOF → 客户端进入 "core lost" 状态 → 定期重连 + 重新 register
- Consumer 端 visual 在 Core 崩溃时显示 "断线"占位（黑屏 + 错误文字），不要 crash
- Producer 崩溃（IPC 断）：Core 释放它持有的所有 canvas + bitmap，给 attached consumer 推 detach 事件
- 句柄安全：DuplicateHandle 给客户端的 NT handle，客户端进程死了 OS 自动回收
- 单帧渲染失败：Core 不 crash，给 producer 返错误码（producer 可以选择 retry 或上报）

### 4.5 多 producer 共写画布（v1.0 默认不开，留接口）

- 用户原话："多个程序同时在一个画布渲染" → 默认走"每个 producer 自己一个画布，layered 显示"
- 真要"多 producer 写同一画布"：owner 显式 grant_write，Core 端帧内串行化（producer A 的 commit_frame 完成后 producer B 才能 commit），不做命令交错合并
- v1.0 第一版可以**先不实现 grant_write**（标记 NotImplemented），spec 留位

### 4.6 监控/安全（v1.0 至少要有 opt-in 机制）

- 风险：恶意 producer create_canvas → attach 全部 monitor → 强制弹屏
- v1.0 最小机制：
  - Consumer 端默认拒绝所有 attach 请求
  - 用户在 monitor UI 里手动允许某个 producer（按 PID / 进程名 / 自签发的 producer-id）
  - 白名单持久化在 monitor 端（不在 Core，因为 Core 不做策略）
- 未来可加：签名机制、用户系统级 toast 确认

## 5. 不要做的事（**避免重蹈历史决策弯路**）

1. **不要换底层** —— D2D 已定，不要试 Vulkan / Skia / wgpu interop。理由见 §2.1。
2. **不要单条命令同步 IPC** —— 性能直接崩。必须 batch + shared memory ringbuffer。
3. **不要 Core 端做命令合并 / 智能合成** —— Core 是机制不是策略。多 producer 写同画布只做时间串行，不做空间合并。
4. **不要保留 in-process DLL 兼容路径** —— v1.0 不向后兼容 v0.7 ABI。旧 main 分支保留作历史快照即可。renderer.dll 完全删除，Core 进程一个。
5. **不要让 Consumer 主动查 / 创建画布** —— Consumer 完全被动，只接受 push。这个语义不能松。
6. **不要忽略 widget UWP 沙盒限制** —— spec 第一个 spike 就是 DComp 跨进程在 UWP 里能不能 work，先验证再投入。
7. **不要忘记 v0.7 已验证的 D2D 实现要复用** —— 命令模式 / cmd_drawing 状态机 / FRAME_HELD / WIC 解码 / bitmap slot ABA 这些代码在 `rust-renderer/src/renderer/` 下都能整体搬到 Core，不要重写。

## 8. 不可妥协的红线

1. **widget 必须能接** — DComp 跨进程 surface 在 UWP 走不通，方案作废
2. **崩溃隔离** — Core 崩了客户端不崩、客户端崩了 Core 不泄漏
3. **60fps 在典型负载** — 命令 batch + shared memory，不能因为跨进程就掉到 30fps
4. **Producer-Owned Canvas 语义不能松** — Consumer 完全被动，画布生命周期跟 owner producer
5. **D2D 保留** — 不允许中途换底层
